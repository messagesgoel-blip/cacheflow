# Task 5.2: BullMQ Cron Worker — Jobs Run Server-Side, Browser Closed

## Overview
This contract defines the BullMQ cron worker system that enables scheduled jobs to run server-side even when browsers are closed. The system consists of a job queue worker and a job execution engine that handles various types of scheduled operations.

## Components

### 1. Scheduled Job Worker (`lib/queue/workers/scheduledJobWorker.ts`)
A BullMQ-based worker that processes scheduled jobs from a Redis-backed queue. Handles job lifecycle including queuing, processing, retries, and cleanup.

### 2. Job Engine (`lib/jobs/jobEngine.ts`)
A job execution engine that defines and executes different types of jobs using registered handlers. Provides a pluggable architecture for different job types.

## Job Types Supported

### sync-file
- **Purpose**: Synchronize files between different storage providers
- **Payload**: `{ sourcePath, destinationPath, providerType, action }`
- **Actions**: upload, download, copy, move

### backup-data
- **Purpose**: Backup data to specified destinations
- **Payload**: `{ sourcePaths, destinationPath, providerType, backupName }`

### cleanup-temp-files
- **Purpose**: Clean up temporary files based on age
- **Payload**: `{ tempDirectories, maxAgeHours }`

### refresh-token
- **Purpose**: Refresh authentication tokens for providers
- **Payload**: `{ providerType }`

## API Interface

### ScheduledJobWorker Class
```typescript
class ScheduledJobWorker {
  constructor(queueName: string = 'scheduled-jobs')
  addJob(
    jobType: string, 
    payload: any, 
    userId?: string,
    opts?: {
      delay?: number;
      repeat?: {
        cron?: string;
        every?: number;
        timezone?: string;
      };
      jobId?: string;
    }
  ): Promise<Job>
  getJob(jobId: string): Promise<Job | null>
  removeJob(jobId: string): Promise<boolean>
  getQueueMetrics(): Promise<{
    waiting: number;
    active: number;
    completed: number;
    failed: number;
    delayed: number;
  }>
  close(): Promise<void>
}
```

## Error Codes

| Code | Description |
|------|-------------|
| `JOB_NOT_FOUND` | Scheduled job with given ID does not exist |
| `JOB_EXECUTION_FAILED` | Job execution encountered an error |
| `INVALID_JOB_TYPE` | No handler registered for the specified job type |
| `PROVIDER_NOT_FOUND` | Storage provider not found or not configured |
| `TOKEN_REFRESH_FAILED` | Failed to refresh authentication tokens |

## Edge Cases

1. **Browser Closed**: Jobs continue to run server-side regardless of browser state
2. **Network Disruption**: Jobs are persisted in Redis and retried with exponential backoff
3. **Concurrent Execution**: Worker handles up to 5 concurrent jobs by default
4. **Token Expiry**: Token refresh jobs ensure continued access to providers
5. **Resource Cleanup**: Completed jobs are removed after 1 hour, failed jobs after 24 hours

## Example Payloads

### Schedule a file sync job
```json
{
  "jobType": "sync-file",
  "payload": {
    "sourcePath": "/folder/file.txt",
    "destinationPath": "/backup/file.txt",
    "providerType": "google",
    "action": "copy"
  },
  "userId": "user-123",
  "opts": {
    "repeat": {
      "cron": "0 2 * * *"  // Daily at 2 AM
    }
  }
}
```

### Schedule a token refresh job
```json
{
  "jobType": "refresh-token",
  "payload": {
    "providerType": "google"
  },
  "userId": "user-123",
  "opts": {
    "repeat": {
      "cron": "0 1 * * *"  // Daily at 1 AM
    }
  }
}
```

## Redis Configuration
- **Database**: Workers namespace (DB index 4)
- **Queue Name**: 'scheduled-jobs' by default
- **Retry Policy**: 3 attempts with exponential backoff starting at 2 seconds
- **Retention**: Completed jobs kept for 1 hour, failed jobs for 24 hours

## Security Considerations
- All job data is processed server-side with no client-side dependencies
- Authentication tokens are managed through secure token vault
- Job payloads containing sensitive data should be encrypted
- User context is maintained through userId in job context
