# Contract: 3.9 — Server streams source → CacheFlow → target without disk write

## Overview

Task 3.9 introduces `streamTransfer`, a server-side function that pipes file
bytes directly from a source `ProviderAdapter`'s `downloadStream` into a
target `ProviderAdapter`'s `uploadStream`.  No bytes are buffered to the
server filesystem at any point — the stream flows entirely through Node.js
memory via a `PassThrough` pipe.

**Gate:** ZERODISK-1

---

## File Created

| File | Role |
|------|------|
| `lib/transfers/streamTransfer.ts` | Core streaming transfer — `streamTransfer()` |

---

## Module path

```
lib/transfers/streamTransfer.ts
```

This is a pure TypeScript library module.  It is not an HTTP endpoint; it is
called by queue workers (task 3.10) or any server-side route that needs to
move a file between two providers without writing to disk.

---

## Function signature

```ts
streamTransfer(opts: StreamTransferOptions): Promise<StreamTransferResult>
```

### `StreamTransferOptions`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `context` | `ProviderOperationContext` | Yes | `requestId`, `userId`, optional `abortSignal` |
| `sourceAdapter` | `ProviderAdapter` | Yes | Adapter that provides `downloadStream` |
| `sourceAuth` | `ProviderAuthState` | Yes | Access credentials for the source provider |
| `sourceFileId` | `string` | Yes | File identifier on the source provider |
| `targetAdapter` | `ProviderAdapter` | Yes | Adapter that provides `uploadStream` |
| `targetAuth` | `ProviderAuthState` | Yes | Access credentials for the target provider |
| `targetParentId` | `string \| undefined` | No | Destination folder on target (root if omitted) |
| `targetFileName` | `string \| undefined` | No | Override file name; defaults to source file name |
| `contentType` | `string \| undefined` | No | MIME type hint; falls back to source `mimeType`, then `application/octet-stream` |
| `onProgress` | `(p: StreamTransferProgress) => void` | No | Optional per-chunk progress callback |

### `StreamTransferProgress`

| Field | Type | Description |
|-------|------|-------------|
| `transferredBytes` | `number` | Bytes that have passed through the pipe so far |
| `totalBytes` | `number` | Total expected bytes from source metadata |
| `percentage` | `number` | Integer 0–100 |

### `StreamTransferResult`

| Field | Type | Description |
|-------|------|-------------|
| `targetFile` | `ProviderFile` | Canonical file record returned by the target adapter |
| `transferredBytes` | `number` | Total bytes that flowed through the pipe |
| `durationMs` | `number` | Wall-clock transfer duration in milliseconds |

## Internal pipe flow

```
sourceAdapter.downloadStream(sourceFileId)
    │
    │  Node.js Readable
    ▼
PassThrough (byte-count tracker, no buffering)
    │
    │  Node.js Readable
    ▼
targetAdapter.uploadStream({ stream: pass, contentLength, … })
    │
    ▼
StreamTransferResult { targetFile, transferredBytes, durationMs }
```

The `PassThrough` transforms nothing; it exists solely so the `data` event
listener can count bytes for progress reporting without interfering with flow
control.

For files whose `contentLength` is ≥ 50 MiB (`STREAM_TRANSFER_CHUNKED_THRESHOLD`),
the known size is passed as `contentLength` to `uploadStream` so provider
adapters that internally use resumable upload sessions can opt into that path.
The pipe itself is always a single uninterrupted stream.

## Error codes

| `AppError.code` | Condition |
|-----------------|-----------|
| `VALIDATION_FAILED` | `context.requestId`, `context.userId`, `sourceFileId`, `sourceAuth.accessToken`, or `targetAuth.accessToken` is missing; `sourceAdapter` or `targetAdapter` is falsy |
| `PROVIDER_UNAVAILABLE` | `sourceAdapter.downloadStream` threw or rejected (source unreachable, auth expired) |
| `TRANSFER_FAILED` | `targetAdapter.uploadStream` threw or rejected (quota exceeded, write error, network) |
| `INTERNAL_ERROR` | Unexpected pipe-level error not caught by the above cases |

All errors are `AppError` instances.  `PROVIDER_UNAVAILABLE` is marked
`retryable: true`; `TRANSFER_FAILED` is `retryable: false`.

## Exported constant

```ts
export { STREAM_TRANSFER_CHUNKED_THRESHOLD }  // 52_428_800 (50 MiB)
```

Callers that need to branch on file size before calling `streamTransfer` can
import this constant to stay consistent with the function's own threshold.

## Edge cases

1. **Unknown file size**: When `contentLength` is absent from the download
   response **and** `sourceFile.size` is 0, `totalBytes` is treated as 0.
   Progress callbacks receive `percentage: 0` on each tick (cannot compute
   a meaningful percentage).  A final tick of 100 % is always emitted on
   successful completion.

2. **AbortSignal cancellation**: `context.abortSignal` is threaded into both
   `downloadStream` and `uploadStream` requests via `ProviderOperationContext`.
   Mid-transfer cancellation surfaces as an `AbortError` from the provider
   adapter, which propagates through the PassThrough and is re-thrown as
   `TRANSFER_FAILED`.

3. **Source stream error after upload starts**: If the `downloadStream` emits
   an `error` event after piping has begun, the PassThrough is destroyed with
   that error, which causes the in-flight `uploadStream` call to reject.  The
   error is surfaced to the caller as `TRANSFER_FAILED`.

4. **Same-provider transfer**: The function imposes no restriction on
   `sourceAdapter === targetAdapter`; a same-provider copy goes through the
   regular download/upload stream path.  Callers that want a more efficient
   server-side copy should use `ProviderAdapter.copyFile` instead.

5. **Memory usage**: No chunk buffering occurs.  Peak heap usage during a
   transfer is bounded by the PassThrough's Node.js internal high-water mark
   (default 16 KiB for object mode, 64 KiB for binary), not by file size.

6. **Upload stream cleanup on failure**: When `uploadStream` rejects, both
   the `downloadStream` and `pass` are destroyed to release the underlying
   HTTP connections and prevent leaks.

7. **fileName override**: If `targetFileName` is omitted, the name is taken
   from `downloadResp.file.name` (resolved at call time), not from the
   `sourceFileId` string.  Callers that need a stable name across retries
   should pass `targetFileName` explicitly.

## Example payload

```ts
import { streamTransfer } from 'lib/transfers/streamTransfer'
import { googleDriveAdapter } from 'lib/providers/googleDrive'
import { dropboxAdapter } from 'lib/providers/dropbox'

const result = await streamTransfer({
  context: {
    requestId: 'req-abc123',
    userId: 'user-42',
  },
  sourceAdapter: googleDriveAdapter,
  sourceAuth: {
    accountId: 'user@example.com',
    accessToken: 'ya29.google_access_token',
  },
  sourceFileId: '1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms',

  targetAdapter: dropboxAdapter,
  targetAuth: {
    accountId: 'user@example.com',
    accessToken: 'sl.dropbox_access_token',
  },
  targetParentId: '/Documents',
  targetFileName: 'backup-2026-03-04.zip',
  contentType: 'application/zip',

  onProgress: ({ transferredBytes, totalBytes, percentage }) => {
    console.log(`${percentage}% — ${transferredBytes}/${totalBytes} bytes`)
  },
})

// result:
// {
//   targetFile: {
//     id: 'id:abc123def456',
//     name: 'backup-2026-03-04.zip',
//     isFolder: false,
//     size: 104857600,
//     mimeType: 'application/zip',
//   },
//   transferredBytes: 104857600,
//   durationMs: 14230,
// }
```
