# Contract: 3.10 — BullMQ background job queue for async transfers

## Overview

Task 3.10 introduces the canonical shared-library BullMQ queue and worker for
background file transfers. These modules live in `lib/queue/` and are consumed
by any process that needs to enqueue or process async transfer jobs — primarily
the worker process and (for enqueueing only) the Next.js API routes.

**Gates:** TRANSFER-1, SSE-1

---

## Files Created

| File | Role |
|------|------|
| `lib/queue/transferQueue.ts` | BullMQ Queue instance, job data/result types, management helpers |
| `lib/queue/workers/transferWorker.ts` | BullMQ Worker — pulls jobs, runs `streamTransfer`, publishes SSE progress |

---

## Module paths

```
lib/queue/transferQueue.ts
lib/queue/workers/transferWorker.ts
```

These are pure TypeScript library modules, not HTTP endpoints.

## Job data shape — `TransferJobData`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `userId` | `string` | Yes | Owner of the job; used for SSE routing and access control |
| `sourceProvider` | `string` | Yes | Provider ID string (e.g. `"google"`, `"dropbox"`) |
| `destProvider` | `string` | Yes | Provider ID string for the destination |
| `fileId` | `string` | Yes | File identifier on the source provider |
| `fileName` | `string` | Yes | Display file name |
| `fileSize` | `number` | Yes | File size in bytes |
| `sourceFolderId` | `string \| undefined` | No | Source folder ID (`undefined` = source root) |
| `destFolderId` | `string \| undefined` | No | Destination folder ID (`undefined` = destination root) |
| `operation` | `"copy" \| "move" \| "upload" \| "download"` | Yes | Semantic operation type |

---

## Job result shape — `TransferJobResult`

| Field | Type | Description |
|-------|------|-------------|
| `success` | `boolean` | Whether the transfer completed without error |
| `jobId` | `string` | BullMQ job ID |
| `fileName` | `string` | Name of the transferred file |
| `fileSize` | `number` | Actual bytes transferred |
| `duration` | `number` | Wall-clock transfer duration in milliseconds |
| `error` | `string \| undefined` | Error message (only on failure) |
| `retryable` | `boolean \| undefined` | Whether BullMQ should retry the job |

## Job ID format

```
transfer-{userId}-{unixMs}
```

Example: `transfer-user-abc123-1741123456789`

## Queue configuration

| Setting | Value | Notes |
|---------|-------|-------|
| `attempts` | 3 | Automatic retries on failure |
| `backoff.type` | `exponential` | Exponential back-off |
| `backoff.delay` | 1 000 ms | Base delay for first retry |
| `removeOnComplete.age` | 3 600 s (1 h) | Completed job TTL |
| `removeOnComplete.count` | 1 000 | Max retained completed entries |
| `removeOnFail.age` | 86 400 s (24 h) | Failed job TTL for debugging |

---

## Worker configuration

| Setting | Value | Notes |
|---------|-------|-------|
| `concurrency` | 5 | Up to 5 simultaneous transfers |
| `limiter.max` | 10 | Max 10 jobs started per `limiter.duration` |
| `limiter.duration` | 1 000 ms | Rate-limit window |

---

## Priority

Large files (≥ 50 MiB) receive BullMQ priority **5**; all others receive
priority **10**.  Lower BullMQ priority number = higher processing urgency.

---

## SSE progress events

The worker publishes three event types via `progressBridge`:

### `progress` (during transfer)

Published on every `onProgress` tick from `streamTransfer`.

```json
{
  "jobId": "transfer-user123-1741123456789",
  "jobType": "transfer",
  "userId": "user123",
  "progress": 42,
  "status": "progress",
  "timestamp": 1741123460000,
  "data": {
    "userId": "user123",
    "sourceProvider": "google",
    "destProvider": "dropbox",
    "fileId": "1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms",
    "fileName": "report.pdf",
    "fileSize": 2097152,
    "operation": "copy"
  }
}
```

### `completed` (on success)

```json
{
  "jobId": "transfer-user123-1741123456789",
  "jobType": "transfer",
  "userId": "user123",
  "progress": 100,
  "status": "completed",
  "timestamp": 1741123465000,
  "result": {
    "success": true,
    "jobId": "transfer-user123-1741123456789",
    "fileName": "report.pdf",
    "fileSize": 2097152,
    "duration": 5000
  }

```

### `failed` (on error)

```json
{
  "jobId": "transfer-user123-1741123456789",
  "jobType": "transfer",
  "userId": "user123",
  "progress": 0,
  "status": "failed",
```

## Error codes

| `AppError.code` | Condition |
|-----------------|-----------|
| `PROVIDER_UNAVAILABLE` | No adapter registered for `sourceProvider` or `destProvider`; source download failed |
| `UNAUTHORIZED` | Auth resolution not yet wired (placeholder until task 3.11) |
| `TRANSFER_FAILED` | `streamTransfer` upload phase failed (quota, network, etc.) — `retryable: false` |
| `INTERNAL_ERROR` | Unexpected error not caught by the above cases |

Errors from `streamTransfer` are normalized via `AppError.fromUnknown` so the
worker result always carries a typed `AppError` message.

---

## Edge cases

1. **Retry behaviour**: When a job fails with `retryable: true` (e.g.
   `PROVIDER_UNAVAILABLE`), BullMQ will automatically re-queue it up to 3
   times with exponential back-off.  `retryable: false` failures (e.g.
   `TRANSFER_FAILED`) are still retried by BullMQ (up to `attempts: 3`) but
   the `retryable` flag is exposed on the result so callers can decide not
   to surface a "retry" button.

2. **Auth resolution (placeholder)**: `resolveAuth` currently throws
   `UNAUTHORIZED`.  It is a named stub that must be wired to the vault/token
   store in task 3.11.  The interface is kept minimal so the integration is
   a one-function swap.

3. **Adapter registry**: Adapters must be registered via
   `registerProviderAdapter(providerId, adapter)` before the worker processes
   a job that uses that provider.  Missing adapters throw
   `PROVIDER_UNAVAILABLE` immediately (no retry).

4. **Graceful shutdown**: `shutdownWorker()` is registered for both `SIGTERM`
   and `SIGINT`.  It calls `Worker.close()` which waits for in-flight jobs to
   complete before releasing the Redis connection.

5. **Concurrent connections**: The queue and worker each hold their own ioredis
   connection derived from `getRedisClient('workers')`.  BullMQ requires
   `maxRetriesPerRequest: null` on its connections; the canonical client is
   configured for this in `lib/redis/client.ts`.

6. **Zero-disk guarantee**: All bytes flow through `streamTransfer`'s
   PassThrough pipe.  No intermediate files are written to disk.

7. **Progress at 0 before start**: The worker publishes a `progress: 0` SSE
   event immediately after dequeuing a job so the UI can show "transfer
   started" before any bytes move.

## Example payload

### Enqueue (caller)

```ts
import { addTransferJob } from 'lib/queue/transferQueue';

const job = await addTransferJob({
  userId: 'user-abc123',
  sourceProvider: 'google',
  destProvider: 'dropbox',
  fileId: '1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms',
  fileName: 'quarterly-report.pdf',
  fileSize: 2_097_152,
  destFolderId: 'id:dbx_folder_xyz',
  operation: 'copy',
});

console.log(job.id);
// → "transfer-user-abc123-1741123456789"
```

### Worker bootstrap (process entry point)

```ts
import { registerProviderAdapter } from 'lib/queue/workers/transferWorker';
import { googleDriveAdapter } from 'lib/providers/googleDrive';
import { dropboxAdapter } from 'lib/providers/dropbox';

registerProviderAdapter('google', googleDriveAdapter);
registerProviderAdapter('dropbox', dropboxAdapter);

// Worker auto-starts processing on import.
// import 'lib/queue/workers/transferWorker';
```

### Queue stats response

```json
{
  "waiting": 3,
  "active": 2,
  "completed": 147,
  "failed": 1,
  "delayed": 0
}
```
