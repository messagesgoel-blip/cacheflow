# Contract: 3.3 — Every File Operation Produces Tray Entry + Toast

## Overview

Every synchronous file operation (delete, move, rename, download) routed through
`app/api/remotes/[uuid]/actions/*/route.ts` now calls `transferRegistry.record()`
before returning its response. The registry is the server-side source of truth for
tray entries; the frontend polls it (or reads the response header) to display
TransferTray items and trigger toast notifications.

## Files Modified / Created

| File | Role |
|------|------|
| `lib/transfers/transferRegistry.ts` | [CREATE] In-process singleton registry keyed by userId |
| `app/api/remotes/[uuid]/actions/delete/route.ts` | [MODIFY] Records delete tray entry |
| `app/api/remotes/[uuid]/actions/move/route.ts` | [MODIFY] Records move tray entry |
| `app/api/remotes/[uuid]/actions/rename/route.ts` | [MODIFY] Records rename tray entry |
| `app/api/remotes/[uuid]/actions/download/route.ts` | [MODIFY] Records download tray entry |

## transferRegistry API

### `transferRegistry.record(userId, opts): TrayEntry`

Called by each action route immediately before returning its HTTP response.

#### `opts` shape

```ts
interface RecordOptions {
  remoteUuid: string;       // UUID from the route path parameter
  provider: string;         // e.g. "google", "onedrive", "dropbox"
  fileId: string;           // Provider-native file identifier
  fileName?: string;        // Human-readable name (optional; available for Filen)
  operation: 'delete' | 'move' | 'rename' | 'download';
  status: 'completed' | 'failed';
  error?: string;           // Present only when status === 'failed'
  toastMessage: string;     // Short sentence for the toast notification
}
```

#### Returned `TrayEntry` shape

```ts
interface TrayEntry {
  id: string;               // "<operation>-<userId>-<timestamp>"
  remoteUuid: string;
  provider: string;
  fileId: string;
  fileName?: string;
  operation: 'delete' | 'move' | 'rename' | 'download';
  status: 'completed' | 'failed';
  error?: string;
  timestamp: number;        // Unix ms
  userId: string;
  toastMessage: string;
}
```

### Read methods

```ts
transferRegistry.getEntries(userId): TrayEntry[]           // All entries, newest first
transferRegistry.getEntriesSince(userId, since): TrayEntry[] // Entries after timestamp
transferRegistry.dismiss(userId, entryId): void
transferRegistry.clearUser(userId): void
transferRegistry.totalEntries(): number
```

## Action Endpoints — Registry Behaviour

### POST /api/remotes/[uuid]/actions/delete

Request body:
```json
{ "fileId": "<string>", "provider": "<string>" }
```

Registry call (success):
```json
{
  "operation": "delete",
  "status": "completed",
  "toastMessage": "Deleted \"<fileId>\" successfully"
}
```

Registry call (failure):
```json
{
  "operation": "delete",
  "status": "failed",
  "error": "Delete failed (404)",
  "toastMessage": "Failed to delete \"<fileId>\""
}
```

## Error Codes

Existing action route error codes are **unchanged** — the registry record is a
fire-and-forget side-effect; it never alters the HTTP status or response body.

| Condition | HTTP status | Registry status |
|-----------|-------------|-----------------|
| Missing required field (fileId etc.) | 400 | No entry recorded |
| Unsupported provider | 400 | No entry recorded |
| Auth expired after refresh retry | 401 | No entry recorded |
| Provider call returns non-2xx | 4xx/5xx | `failed` entry recorded |
| Provider call returns 2xx | 200 | `completed` entry recorded |

---

## Edge Cases

1. **Missing `accessToken` cookie or JWT decode failure**: `userId` falls back to
   `"anonymous"`. The tray entry is still recorded; the frontend must handle
   the `"anonymous"` key gracefully (e.g. discard, or prompt re-auth).

2. **Providers with unsupported operations** (e.g. Dropbox delete, pCloud move):
   Routes return 400 before the provider call — no registry entry is written,
   consistent with the "no entry for validation errors" rule above.

3. **Auth 401 → refresh → 401**: Both the route and the registry return early with
   `requiresReauth: true` before recording — no misleading `failed` entry is
   created for an auth issue.

4. **Concurrent operations for the same user**: The registry `store` is a `Map`
   protected by single-threaded JavaScript; no locks needed. Entries are prepended
   atomically in a single array slice assignment.

5. **Registry eviction**: Each user is capped at 50 entries (oldest evicted).
   Entries are never persisted across process restarts — the tray is ephemeral.

6. **Download — multi-step providers** (pCloud, Filen, Yandex): The registry entry
   is recorded at the **final** terminal return inside the provider-specific branch,
   not at the intermediate link-fetch step, so the status reflects the actual
   download outcome.

## Example Payload (delete success)

```ts
transferRegistry.record("user-42", {
  remoteUuid: "a1b2c3d4-...",
  provider: "google",
  fileId: "1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms",
  operation: "delete",
  status: "completed",
  toastMessage: "Deleted \"1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms\" successfully",
});
```
