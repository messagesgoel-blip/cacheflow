# Contract: 4.1

## Endpoint or File Path

- `docs/provider-parity.md`
- `lib/providers/ProviderAdapter.interface.ts`

No HTTP endpoint is introduced by this task; the contract is file-based.

## Request / Response Shape

### Canonical checklist shape (code export)

```ts
type ProviderParityCheckId =
  | 'auth_lifecycle'
  | 'file_discovery'
  | 'file_mutation'
  | 'stream_transfer'
  | 'resumable_transfer'

interface ProviderParityCheck {
  id: ProviderParityCheckId
  title: string
  requiredMethods: ReadonlyArray<Exclude<keyof ProviderAdapter, 'descriptor'>>
}

const PROVIDER_PARITY_CHECKLIST: ReadonlyArray<ProviderParityCheck>
```

### Evaluation I/O shape (for parity tests/docs)

```ts
interface ProviderParityEvaluationRequest {
  providerId: string
  checks: ReadonlyArray<ProviderParityCheckId> // must include all 5
}

interface ProviderParityEvaluationResponse {
  providerId: string
  passed: boolean
  results: Array<{
    checkId: ProviderParityCheckId
    passed: boolean
    reason?: string
  }>
}
```

## Error Codes

Parity failures and adapter operation failures use canonical provider error codes:

- `UNAUTHORIZED`
- `TOKEN_EXPIRED`
- `REFRESH_FAILED`
- `RATE_LIMITED`
- `PROVIDER_UNAVAILABLE`
- `QUOTA_EXCEEDED`
- `CHUNK_FAILED`
- `TRANSFER_FAILED`
- `NOT_FOUND`
- `CONFLICT`
- `VALIDATION_FAILED`

## Edge Cases

- Provider passes 4/5 checks: overall parity result is failure.
- Provider lacks native resumable support: must still expose resumable methods and return deterministic canonical errors.
- Provider returns non-canonical errors: parity check fails even if operation appears to succeed.
- Pagination drift (`nextCursor` invalid or `hasMore` inconsistent): fails `file_discovery`.
- Stream code writing to temp disk paths: fails `stream_transfer`.
