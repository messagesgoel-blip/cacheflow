# Contract: 3.14 — Provider health indicators — green means it actually works

## Overview

Task 3.14 introduces a real provider liveness probe.  Rather than returning a
`connected` status based solely on whether a credential row exists in the DB,
this task makes a lightweight, read-only call to each provider's identity
endpoint and surfaces the actual result.

**Gate:** SYNC-1

---

## Files Created

| File | Role |
|------|------|
| `lib/providers/healthCheck.ts` | Provider probe logic — `probeProvider()` and `probeAllProviders()` |
| `web/app/api/connections/health/route.ts` | Next.js App Router GET handler |

---

## Endpoint

```
GET /api/connections/health
```

Requires authentication (JWT in `accessToken` cookie or `Authorization: Bearer <token>` header).

---

## Request

No body.  Auth is read from:
- `accessToken` cookie (set by the Express auth flow), **or**
- `Authorization: Bearer <jwt>` header

---

---

## Response — 401 Unauthorized

```json
{ "success": false, "error": "Authentication required" }
```

Returned when no valid JWT is present.

---

## Response — 502 Bad Gateway

```json
{ "success": false, "error": "Failed to fetch connections" }
```

Returned when the upstream Express API (`/api/remotes`) is unreachable.

---

## `HealthStatus` values

| Value | Meaning | UI color |
|-------|---------|----------|
| `healthy` | Credential accepted, provider reachable | Green |
| `needs_reauth` | 401/403 from provider — token invalid or revoked | Red |
| `degraded` | 5xx, 429, timeout, or network error | Amber |
| `unknown` | No probe implementation for this provider | Grey |

---

## `HealthProbeResult` shape

| Field | Type | Description |
|-------|------|-------------|
| `status` | `HealthStatus` | One of `healthy`, `needs_reauth`, `degraded`, `unknown` |
| `checkedAt` | `string` | ISO-8601 timestamp of probe execution |
| `httpStatus` | `number?` | HTTP status code from the provider (absent for network errors) |
| `message` | `string` | Human-readable probe outcome |
| `latencyMs` | `number` | Wall-clock probe duration in milliseconds |

---

## Per-provider probe endpoints

| Provider | Probe URL | Method |
|----------|-----------|--------|
| `google` | `https://www.googleapis.com/drive/v3/about?fields=user%28emailAddress%29` | GET |
| `onedrive` | `https://graph.microsoft.com/v1.0/me?$select=userPrincipalName` | GET |
| `dropbox` | `https://api.dropboxapi.com/2/users/get_current_account` | POST (null body) |
| `box` | `https://api.box.com/2.0/users/me?fields=id,login` | GET |
| `pcloud` | `https://api.pcloud.com/userinfo?access_token=<token>` | GET (query param) |
| `yandex` | `https://login.yandex.ru/info` | GET (`OAuth` prefix) |
| `webdav`, `vps`, `local`, `filen` | — | `unknown` (no probe) |

---

## Probe timeout

Each provider probe has a **2-second** wall-clock timeout enforced via
`AbortController`.  Timeout results in `status: "degraded"`.

---

## Probe execution model

Probes for all remotes run **concurrently** (`Promise.all`).  Individual
failures do not fail the whole batch.  A disabled remote is returned
immediately with `status: "needs_reauth"` and `message: "Connection is disabled"`
without making a network call.

---

## Edge cases

1. **Disabled remotes**: skipped for network probing; returned as
   `needs_reauth` / `"Connection is disabled"`.
2. **Provider with no probe config**: returns `status: "unknown"` immediately
   (no network call).
3. **Express API unreachable**: the whole request returns HTTP 502
   `"Failed to fetch connections"`.
4. **Empty remotes list**: returns HTTP 200 with `connections: []`.
5. **Expired token in DB**: the probe will return `needs_reauth` when the
   provider rejects the stale token.
6. **QA mock mode**: if the Express backend's proxy returns mock 200 for
   seeded tokens, the health probe will report `healthy`.

---

## Token flow

The route does **not** decrypt tokens directly.  It calls the Express backend's
`POST /api/remotes/:id/proxy` endpoint which decrypts and injects the
`Authorization: Bearer <token>` header server-side before forwarding the
probe request to the provider.
