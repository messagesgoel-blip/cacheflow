# Contract: 3.15 — Rate limit handling layer — per-provider request queue

## Overview

Task 3.15 introduces `lib/providers/rateLimitQueue.ts`, a per-provider in-process
request queue that enforces concurrency limits and minimum dispatch intervals for
all outbound provider API calls.  It also adds a `proxyRequestRateLimited` method
to `lib/apiClient.ts` that routes proxy calls through this queue and handles HTTP
429 responses with automatic back-off.

### `ProviderQueueConfig`

```ts
interface ProviderQueueConfig {
  maxConcurrent: number      // max simultaneous in-flight requests
  minIntervalMs: number      // minimum ms between consecutive dispatches
  queueTimeoutMs?: number    // max wait before RATE_LIMITED error (default 30 000)
}
```

### `RateLimitedCallResult<T>`

```ts
interface RateLimitedCallResult<T> {
  data: T
  wasDelayed: boolean  // true when request waited >5 ms before dispatch
  waitMs: number       // total wall-clock wait in ms before dispatch started
}
```

| Provider | `maxConcurrent` | `minIntervalMs` | Basis |
|----------|----------------|-----------------|-------|
| `google` | 6 | 110 ms | Drive API: ~1 000 req/100 s per user |
| `onedrive` | 4 | 65 ms | Graph API: ~10 000 req/10 min per app |
| `dropbox` | 10 | 50 ms | Published burst: 200 req/user/s |
| `box` | 4 | 65 ms | Box API: ~1 000 req/min per user |
| `pcloud` | 3 | 200 ms | Undocumented; conservative |
| `filen` | 3 | 200 ms | Undocumented; conservative |
| `yandex` | 3 | 200 ms | Undocumented; conservative |

| `vps` | 8 | 0 ms | No external provider limit |

### `RateLimitQueue.enqueue(providerId, fn)`

```ts
enqueue<T>(providerId: ProviderId, fn: () => Promise<T>): Promise<RateLimitedCallResult<T>>
```

Queues `fn` for dispatch according to the provider's concurrency and interval
constraints.  Throws `AppError(RATE_LIMITED)` if the request waits longer than
`queueTimeoutMs`.

### `RateLimitQueue.applyRateLimitBackoff(providerId, retryAfterMs?)`

```ts
applyRateLimitBackoff(providerId: ProviderId, retryAfterMs?: number): void
```

Pauses dispatch for `retryAfterMs` milliseconds (default 60 000).  Monotonically
increases an active pause — never shortens it.

### `RateLimitQueue.configure(providerId, overrides)`

```ts
configure(providerId: ProviderId, overrides: Partial<ProviderQueueConfig>): void
```

Runtime override of queue config.  Takes effect on the next dispatch.

## Method: `apiClient.proxyRequestRateLimited`

### Signature

```ts
proxyRequestRateLimited(
  providerId: ProviderId,
  remoteId: string,
  method: string,
  url: string,
  headers?: Record<string, string>,
  body?: unknown,
): Promise<ApiResponse<unknown> & { rateLimited?: boolean; waitMs?: number }>
```

### Behaviour

1. Calls `getRateLimitQueue().enqueue(providerId, ...)` — request waits if the
   provider queue is at capacity or paused.
2. On HTTP **429**: calls `applyRateLimitBackoff(providerId, retryAfterMs)` and
   returns `success: false, rateLimited: true`.
3. On queue-wait **timeout**: returns `success: false, rateLimited: true`.
4. On any other non-2xx: returns `success: false, error: "HTTP N: ..."`.
5. On success: returns `success: true, data: <json>, waitMs: <number>`.

### Response shape

```jsonc
// Success
{
  "success": true,
  "data": { /* provider response JSON */ },
  "waitMs": 12
}

// Rate limited by provider (HTTP 429)
{
  "success": false,
  "error": "Rate limited by provider \"google\" — retry after 60000 ms",
  "rateLimited": true,
  "waitMs": 0
}

// Queue wait timeout (queueTimeoutMs exceeded before dispatch)
{
  "success": false,
  "error": "Provider \"google\" queue wait exceeded 30000 ms",
  "rateLimited": true
}

// Other HTTP error
{
  "success": false,
  "error": "HTTP 403: Forbidden",
  "waitMs": 5
}
```

## Error codes

| Condition | `ErrorCode` | HTTP equivalent |
|-----------|-------------|-----------------|
| Queue wait timeout | `RATE_LIMITED` | 429 |
| Provider 429 back-off active | `RATE_LIMITED` | 429 |

---

## Edge cases

1. **Concurrent 429s**: `applyRateLimitBackoff` is idempotent — subsequent calls
   during an active pause will not shorten the existing back-off window.
2. **In-flight requests on 429**: already-dispatched requests are not cancelled;
   only future enqueue calls are held.
3. **Unknown provider**: falls back to `maxConcurrent: 4, minIntervalMs: 200`.
4. **`Retry-After` header absent**: defaults to 60 000 ms back-off.
5. **`Retry-After` as HTTP-date**: parsed via `Date.parse`; delta converted to ms.
6. **Queue timeout while paused**: the timeout fires normally and the caller
   receives `RATE_LIMITED` regardless of whether the pause is still active.
7. **`configure()` during active pause**: changes `minIntervalMs`/`maxConcurrent`
   immediately; does **not** affect the active pause timer.

## Example usage

```ts
import { apiClient } from 'lib/apiClient';

const result = await apiClient.proxyRequestRateLimited(
  'google',
  'remote-uuid-abc',
  'GET',
  'https://www.googleapis.com/drive/v3/files',
  { Authorization: 'Bearer <token>' },
);

if (!result.success && result.rateLimited) {
  // Back-off is already applied — caller should surface a retry message to the user
  console.warn('Rate limited, queue paused', result.error);
} else if (result.success) {
  console.log('Files:', result.data);
}
```

```ts
import { getRateLimitQueue } from 'lib/apiClient';

const stats = getRateLimitQueue().getAllStats();
// { google: { inflight: 2, pending: 5, pausedUntil: null, config: { ... } }, ... }
```
