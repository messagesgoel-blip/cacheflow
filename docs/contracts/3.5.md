# Contract: 3.5 — Chunked Upload: Files >50 MB Use Provider Resumable Upload APIs

## Overview

Files larger than 50 MB are uploaded via each provider's native resumable/session
upload API rather than a single-request PUT. Each provider module exposes a
standalone async function that:

1. Opens a provider upload session.
2. Splits the stream into fixed-size chunks.
3. Uploads each chunk with retry on transient errors (429, 5xx).
4. Returns a canonical `ProviderFile` on completion.

## Threshold

All three modules export `CHUNKED_UPLOAD_THRESHOLD = 52_428_800` (50 MiB).
Callers must route to the chunked path when `contentLength > CHUNKED_UPLOAD_THRESHOLD`.

## Google Drive

### Provider API

```
POST https://www.googleapis.com/upload/drive/v3/files?uploadType=resumable
PUT  <uploadUrl>  (repeated per chunk)
```

### Session Initiation — Request

```http
POST https://www.googleapis.com/upload/drive/v3/files?uploadType=resumable
Authorization: Bearer <accessToken>
Content-Type: application/json
X-Upload-Content-Type: <mimeType>
X-Upload-Content-Length: <totalBytes>

{
  "name": "<fileName>",
  "parents": ["<parentId>"]   // omit if uploading to root
}
```

### Session Initiation — Response

```
HTTP/1.1 200 OK
Location: https://www.googleapis.com/upload/drive/v3/files?uploadType=resumable&upload_id=<id>
```

### Chunk Upload — Request

```http
PUT <uploadUrl>
Content-Range: bytes <start>-<end>/<total>
Content-Length: <chunkBytes>
Content-Type: application/octet-stream

<binary chunk data>
```

### Chunk Upload — Response codes

| Status | Meaning |
|--------|---------|
| `308 Resume Incomplete` | Chunk accepted; `Range` header reports committed bytes |
| `200 OK` / `201 Created` | Final chunk accepted; body is the Drive file metadata JSON |

### Default chunk size

`8 MiB` (exported as `GOOGLE_DEFAULT_CHUNK_SIZE`). Must be a multiple of 256 KiB per Google's requirement.

### Function signature

```ts
googleDriveChunkedUpload(opts: GoogleDriveChunkedUploadOptions): Promise<GoogleDriveChunkedUploadResult>
```

```ts
interface GoogleDriveChunkedUploadOptions {
  context: ProviderOperationContext   // requestId, userId, abortSignal
  auth: ProviderAuthState             // accessToken
  parentId?: string
  fileName: string
  contentType?: string                // default: 'application/octet-stream'
  contentLength: number               // required — passed to X-Upload-Content-Length
  stream: Readable
  chunkSize?: number                  // default: GOOGLE_DEFAULT_CHUNK_SIZE (8 MiB)
  onProgress?: (p: { uploadedBytes: number; totalBytes: number; percentage: number }) => void
}

interface GoogleDriveChunkedUploadResult {
  file: ProviderFile
  uploadedBytes: number
}
```

## Dropbox

### Provider API

```
POST https://content.dropboxapi.com/2/files/upload_session/start
POST https://content.dropboxapi.com/2/files/upload_session/append_v2   (repeated per non-final chunk)
POST https://content.dropboxapi.com/2/files/upload_session/finish       (final chunk + commit)
```

### Session Start — Request

```http
POST https://content.dropboxapi.com/2/files/upload_session/start
Authorization: Bearer <accessToken>
Content-Type: application/octet-stream
Dropbox-API-Arg: {"close": false}

(empty body)
```

### Session Start — Response

```json
{ "session_id": "1234faaf0678bcde" }
```

### Append — Request

```http
POST https://content.dropboxapi.com/2/files/upload_session/append_v2
Authorization: Bearer <accessToken>
Content-Type: application/octet-stream
Dropbox-API-Arg: {"cursor": {"session_id": "<id>", "offset": <n>}, "close": false}

<binary chunk data>
```

Successful response: `HTTP 200` with empty body.

### Finish — Request

```http
POST https://content.dropboxapi.com/2/files/upload_session/finish
Authorization: Bearer <accessToken>
Content-Type: application/octet-stream
Dropbox-API-Arg: {
  "cursor": {"session_id": "<id>", "offset": <finalOffset>},
  "commit": {"path": "/<parentId>/<fileName>", "mode": "add", "autorename": true}
}

<final binary chunk data>
```

Successful response: `HTTP 200` with the Dropbox file metadata JSON.

### Default chunk size

`150 MiB` (exported as `DROPBOX_DEFAULT_CHUNK_SIZE`). Dropbox maximum per append call is 150 MiB.

### Function signature


## Error Codes

All functions throw `AppError` with the following codes:

| Code | Condition |
|------|-----------|
| `UNAUTHORIZED` | `accessToken` rejected by provider (HTTP 401) |
| `RATE_LIMITED` | Provider returns HTTP 429 |
| `PROVIDER_UNAVAILABLE` | Session initiation failed with non-auth 4xx or 5xx |
| `CHUNK_FAILED` | Individual chunk PUT rejected after all retries |
| `TRANSFER_FAILED` | Upload incomplete after all chunks sent, or finish call failed |
| `VALIDATION_FAILED` | `contentLength` not provided, or stream produced no data |
| `INTERNAL_ERROR` | Provider response missing expected session URL or session ID |

Transient errors (429, 5xx) are retried up to **3 times** with exponential back-off (1s, 2s, 4s).

---

## Edge Cases

1. **Empty stream**: All three functions throw `AppError(VALIDATION_FAILED)` if the buffered stream produces zero bytes.

2. **AbortSignal cancellation**: `context.abortSignal` is forwarded to every `fetch` call. Mid-upload cancellation surfaces as a native `AbortError` from the underlying `fetch`.

3. **Stream larger than `contentLength`**: The stream is fully buffered and chunked from the buffer; `contentLength` is used only for the provider session headers and progress calculation. Callers must pass an accurate value.

4. **Single-chunk files >50 MB**: All three providers handle files that fit in one chunk — the loop executes once (Google: 200/201 on first PUT; OneDrive: 200/201 on first PUT; Dropbox: `start` + `finish` with no `append`).

5. **Chunk size constraints**: Google requires multiples of 256 KiB. OneDrive requires multiples of 320 KiB. Dropbox max is 150 MiB. Callers that override `chunkSize` are responsible for alignment.

6. **Session expiry**: Upload sessions are not automatically re-created. Callers that resume after long pauses should check provider session expiry (`expiresAt` for OneDrive) and re-initiate if needed.

7. **OneDrive final-chunk detection**: The function relies on a 200/201 response status to identify the final chunk. If the server returns 202 for the last chunk (non-standard), the function falls back to `queryUploadStatus` and surfaces a `TRANSFER_FAILED` error if the upload is incomplete.

8. **Dropbox parentId as path**: For Dropbox, `parentId` is treated as a path prefix (e.g. `/photos`), matching the convention in `dropbox.ts`. The commit path becomes `${parentId}/${fileName}`.

## Example Payload (Google Drive — 100 MiB file, 2 chunks of 8 MiB each truncated for brevity)

```ts
const result = await googleDriveChunkedUpload({
  context: { requestId: 'req-abc', userId: 'user-42' },
  auth: { accountId: 'user@example.com', accessToken: 'ya29.xxx' },
  parentId: '1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms',
  fileName: 'archive.zip',
  contentType: 'application/zip',
  contentLength: 104857600,
  stream: fs.createReadStream('/tmp/archive.zip'),
  chunkSize: 8 * 1024 * 1024,
  onProgress: ({ uploadedBytes, percentage }) =>
    console.log(`${percentage}% (${uploadedBytes} bytes)`),
})

//   checksum: 'md5hex...'
// }
```
