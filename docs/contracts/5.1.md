# Task Contract 5.1: Scheduled job data model and Schedules management UI

## Data Model

### ScheduledJob Model
- **id**: string (primary key)
- **name**: string (job display name)
- **jobType**: string (type of job, e.g., 'backup', 'sync', 'cleanup')
- **cronExpression**: string (CRON expression for scheduling)
- **enabled**: boolean (whether the job is currently enabled, default: true)
- **lastRunAt**: DateTime? (timestamp of last execution)
- **nextRunAt**: DateTime? (timestamp of next scheduled execution)
- **createdAt**: DateTime (creation timestamp)
- **updatedAt**: DateTime (last update timestamp)

## API Endpoints

### GET /api/jobs
- **Purpose**: Retrieve all scheduled jobs or a specific job
- **Query Parameters**:
  - `id` (optional): Job ID to retrieve specific job
- **Response**:
  - Success: 200 with array of jobs (or single job if ID specified)
  - Error: 404 if job not found by ID
  - Error: 500 for server errors
- **Example Request**: `GET /api/jobs?id=job_1234567890`
- **Example Response**: 
```json
{
  "id": "job_1234567890",
  "name": "Daily Backup",
  "jobType": "backup",
  "cronExpression": "0 2 * * *",
  "enabled": true,
  "lastRunAt": "2023-01-01T02:00:00Z",
  "nextRunAt": "2023-01-02T02:00:00Z",
  "createdAt": "2023-01-01T00:00:00Z",
  "updatedAt": "2023-01-01T00:00:00Z"
}
```

### POST /api/jobs
- **Purpose**: Create a new scheduled job
- **Request Body**:
  - `name` (required): Job display name
  - `jobType` (required): Type of job
  - `cronExpression` (required): CRON expression for scheduling
  - `enabled` (optional): Whether job should be enabled (default: true)
- **Response**:
  - Success: 201 Created with created job object
  - Error: 400 for missing required fields
  - Error: 500 for server errors
- **Example Request**:
```json
{
  "name": "Weekly Sync",
  "jobType": "sync",
  "cronExpression": "0 3 * * 0",
  "enabled": true
}
```

### PUT /api/jobs?id={id}
- **Purpose**: Update an existing scheduled job
- **Query Parameters**:
  - `id` (required): Job ID to update
- **Request Body** (all optional):
  - `name`: New job display name
  - `jobType`: New job type
  - `cronExpression`: New CRON expression
  - `enabled`: Whether job should be enabled
- **Response**:
  - Success: 200 with updated job object
  - Error: 400 for missing job ID
  - Error: 404 if job not found
  - Error: 500 for server errors
- **Example Request**:
```json
{
  "name": "Updated Weekly Sync",
  "enabled": false
}
```

### DELETE /api/jobs?id={id}
- **Purpose**: Delete a scheduled job
- **Query Parameters**:
  - `id` (required): Job ID to delete
- **Response**:
  - Success: 200 with success message
  - Error: 400 for missing job ID
  - Error: 404 if job not found
  - Error: 500 for server errors

## Error Codes

- **400 Bad Request**: Missing required fields or invalid parameters
- **404 Not Found**: Job with specified ID does not exist
- **500 Internal Server Error**: Unexpected server error during operation

## Edge Cases

- Creating a job with invalid CRON expression should be validated by the calling service
- Updating a job with only some fields should preserve unchanged fields
- Disabling a job should prevent it from executing but retain its configuration
- Jobs with `enabled: false` should still be returned in GET requests but won't execute
- Cron expressions should follow standard unix CRON format
