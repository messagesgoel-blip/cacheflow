# Contract: 0.3 - Prisma Schema Baseline

## Overview
Database schema definition for CacheFlow covering authentication vault, file transfers, and rate limiting.

## File Path
- `/prisma/schema.prisma` - Prisma ORM schema
- `/docs/architecture/data-model.md` - Data model documentation

## Authentication (AUTH-2)

**Token Vault:**
- Maximum 3 accounts per provider per user
- AES-256-GCM encrypted credentials
- Multi-account with accountOrder (1-3)
- One default account per provider

**Fields:**
- `provider`: VARCHAR(50) - Provider ID
- `accountLabel`: VARCHAR(50) - Account name
- `accountOrder`: INTEGER - Display order (1-3)
- `isDefault`: BOOLEAN - Default account flag
- `encryptedCredentials`: BYTEA - Encrypted JSON

## Transfers (TRANSFER-1)

**Transfer Table:**
- Persistent history for completed/failed transfers
- Active transfers managed via BullMQ/Redis

**Status Values:**
- `pending` - Waiting in queue
- `running` - Currently processing
- `completed` - Successfully finished
- `failed` - Failed with error
- `cancelled` - User cancelled

**Operations:**
- `copy` - Copy between providers
- `move` - Move between providers
- `upload` - Upload to provider
- `download` - Download from provider

## Rate Limiting (TRANSFER-1)

**RateLimit Table:**
- Per-user, per-provider, per-action tracking
- Window-based (typically 1 hour)

## Error Codes

| Code | Description |
|------|-------------|
| AUTH_001 | Invalid credentials |
| AUTH_002 | Token expired |
| AUTH_003 | 2FA required |
| AUTH_004 | Account locked |
| TRANSFER_001 | Transfer failed |
| TRANSFER_002 | Rate limit exceeded |
| VAULT_001 | Vault decryption failed |
| VAULT_002 | Max accounts reached |

## Edge Cases

1. **Multi-Account Limits**: Prevent adding >3 accounts per provider
2. **Tenant Isolation**: All queries must include tenant_id check
3. **Token Expiration**: Cleanup expired tokens periodically
4. **Transfer Resume**: Use Transfer table for persistence across restarts
5. **Conflict Resolution**: Track both local and cloud versions
6. **Rate Limit Windows**: Sliding window implementation

## Dependencies

- **AUTH-2**: Requires ProviderAdapter interface (0.1)
- **TRANSFER-1**: Requires BullMQ queue (3.10)
- **VAULT-1**: Requires encryption utilities

## Notes

- Active transfers use BullMQ/Redis for real-time processing
- Transfer table provides persistent history and analytics
- All sensitive data encrypted at rest (AES-256-GCM)
- Tenant isolation enforced via FK + application checks
