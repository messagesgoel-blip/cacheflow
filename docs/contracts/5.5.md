# VAULT-1: Vault Data Model and Enable/Disable API

## Data Model

### Vault Schema
```sql
TABLE "Vault" (
    "id" TEXT NOT NULL,           -- Unique vault identifier
    "userId" TEXT NOT NULL,       -- Foreign key to User
    "isEnabled" BOOLEAN NOT NULL DEFAULT true,  -- Whether vault is enabled
    "encryptedData" TEXT NOT NULL,              -- Encrypted vault contents
    "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updatedAt" TIMESTAMP(3) NOT NULL
)
```

## API Endpoints

### GET /api/vault
Fetch the current vault status for authenticated user.

#### Request
- Method: GET
- Headers: Authorization (session cookie)
- Body: None

#### Response
- Success: 200 OK
```json
{
  "isEnabled": boolean,
  "createdAt": string,
  "updatedAt": string
}
```
- If no vault exists: 200 OK with `{ "isEnabled": false }`
- Unauthorized: 401 Unauthorized
- Server Error: 500 Internal Server Error

### POST /api/vault
Enable or disable the vault for authenticated user.

#### Request
- Method: POST
- Headers: Authorization (session cookie), Content-Type: application/json
- Body:
```json
{
  "isEnabled": boolean
}
```

#### Response
- Success: 200 OK
```json
{
  "success": true,
  "isEnabled": boolean
}
```
- Invalid request: 400 Bad Request
- Unauthorized: 401 Unauthorized
- Server Error: 500 Internal Server Error

## Error Codes

- 400: Invalid request body (isEnabled must be boolean)
- 401: Unauthorized (no valid session)
- 500: Internal server error

## Edge Cases

- User has no existing vault record: Creates new vault with default values
- Vault upsert operation handles both create and update scenarios
- Empty encryptedData initialized as '{}' for new vaults
- Foreign key constraint ensures vault is tied to existing user

## Example Payload

### Request
```json
{
  "isEnabled": true
}
```

### Response
```json
{
  "success": true,
  "isEnabled": true
}
```
