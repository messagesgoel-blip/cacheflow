# Contract: 6.2 — Transfer Logs Supervised Experience

## Overview

Persistent terminal view renders real-time BullMQ and worker events over SSE with zero polling.

## Endpoint

**Primary path:** `/api/jobs/logs?jobId=...`

**Method:** GET (SSE stream)

**Description:** Server-Sent Events stream for real-time worker logs plus authoritative progress/completion updates for a single job.

**Fallback path (legacy):** `/api/transfers/[id]/progress`

## Request Shape

### Primary Path Query Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `jobId` | string | Yes | Job ID to monitor |

### Legacy Fallback Path Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | string | Yes | Transfer or job identifier used by `/api/transfers/[id]/progress` |

### Authentication

- Required cookie: `accessToken` (HttpOnly). Browsers send it automatically with same-origin SSE requests; non-browser clients must include it in the `Cookie` header.
- The same cookie requirement applies to both `/api/jobs/logs` and the legacy fallback `/api/transfers/[id]/progress`.
- Requests without a valid `accessToken` cookie are rejected with `401`.

## Response Shape

### Success (200 OK - SSE)

```typescript
interface JobLogsStreamEvents {
  connected: {
    jobId: string;
    userId: string;
  };
  log: {
    jobId: string;
    jobType: 'transfer';
    userId: string;
    level: 'info' | 'warn' | 'error' | 'debug';
    message: string;
    timestamp: number;
    data?: Record<string, unknown>;
  };
  progress: {
    jobId: string;
    jobType: 'transfer' | 'rate_limit';
    userId: string;
    progress: number | Record<string, unknown>;
    status: 'progress' | 'completed' | 'failed';
    timestamp: number;
    data?: Record<string, unknown>;
    result?: Record<string, unknown>;
    error?: string;
  };
  done: {
    status: 'completed' | 'failed';
    error?: string;
  };
}
```

## Edge Cases

1. **Jobs complete:**
   - Stream emits `done` with `{status:"completed"}` and then closes
2. **Job fails:** Stream emits `done` with `{status:"failed", error}` and then closes
3. **Connection drops:** Client reconnects automatically
4. **Expired credentials / invalid auth:** request is rejected with `401`
5. **Server misconfiguration:** request is rejected with `500` if `JWT_SECRET` is unset

## Example Payload

```text
event: connected
data: {"jobId":"transfer-123","userId":"user-456"}

event: log
data: {"jobId":"transfer-123","jobType":"transfer","userId":"user-456","level":"info","message":"Starting transfer","timestamp":1700000000000}

```

## Implementation Notes

- **Primary endpoint**: `/api/jobs/logs?jobId=xxx` - Worker log stream used by `JobLogPanel`
- **Fallback endpoint**: `/api/transfers/[id]/progress` - Legacy transfer progress stream, retained only as fallback
- **Auth requirement**: consumers must open the SSE stream with the authenticated HttpOnly `accessToken` cookie present
- Progress bridge extended with `publishWorkerLog()` and `publishTransferLog()` methods
- Progress emitter now exposes `onJobLogs()` for subscribing to worker log events
- Transfer worker publishes log events at key lifecycle points:
  - Start: "Starting transfer: {operation} {filename} ({size} bytes) {source} -> {target}"
  - Progress: via progress events
  - Success: "Transfer completed successfully in {duration}ms"
  - Failure: "Transfer failed: {error}"
- Log events include: jobId, jobType, userId, level (info/warn/error/debug), message, timestamp, data
- **SSE Contract**: The endpoint emits:
  - `connected`: Initial connection with {jobId, userId}
  - `log`: Worker log entry with full WorkerLogEntry payload
  - `progress`: Progress update from the job
  - `done`: Terminal event with {status: 'completed' | 'failed', error?} - signals stream end
- **Scope**: This implementation covers transfer jobs. Scheduled job logs are not part of the supported contract until `scheduledJobWorker` is integrated end-to-end.
