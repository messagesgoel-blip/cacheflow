# Contract: 6.2 — Transfer Logs Supervised Experience

## Overview

Persistent terminal view renders real-time BullMQ and worker events over SSE with zero polling.

## Endpoint

**Path:** `/api/transfers/[id]/progress`

**Method:** GET (SSE stream)

**Description:** Server-Sent Events stream for real-time job progress updates.

## Request Shape

### Query Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | string | Yes | Transfer job ID to monitor |

## Response Shape

### Success (200 OK - SSE)

```typescript
interface JobProgressEvent {
  type: 'progress' | 'complete' | 'error' | 'heartbeat';
  jobId: string;
  payload?: {
    progress: number;
    total?: number;
    message?: string;
    timestamp: string;
  };
  error?: string;
}
```

## Edge Cases

1. **Jobs complete:**
   - Stream emits `complete` event with final stats
2. **Job fails:** Stream emits `error` event with message
3. **Connection drops:** Client reconnects automatically
4. **Expired credentials:** Stream emits `error` with 401

## Example Payload

```
event: connected
data: {"jobId":"transfer-123","userId":"user-456"}

event: progress
data: {"type":"progress","jobId":"transfer-123","payload":{"progress":50,"total":100,"message":"Transferring..."}}

event: done
data: {"status":"completed"}
```

## Implementation Notes

- **Primary endpoint**: `/api/jobs/logs?jobId=xxx` - Worker log events (LOGS-1)
- **Fallback endpoint**: `/api/transfers/[id]/progress` - Transfer progress
- Progress bridge extended with `publishWorkerLog()` and `publishTransferLog()` methods
- Progress emitter now exposes `onJobLogs()` for subscribing to worker log events
- Transfer worker publishes log events at key lifecycle points:
  - Start: "Starting transfer: {operation} {filename} ({size} bytes) {source} -> {target}"
  - Progress: via progress events
  - Success: "Transfer completed successfully in {duration}ms"
  - Failure: "Transfer failed: {error}"
- Log events include: jobId, jobType, userId, level (info/warn/error/debug), message, timestamp, data
- **SSE Contract**: The endpoint emits:
  - `connected`: Initial connection with {jobId, userId}
  - `log`: Worker log entry with full WorkerLogEntry payload
  - `progress`: Progress update from the job
  - `done`: Terminal event with {status: 'completed' | 'failed', error?} - signals stream end
- **Scope**: This implementation covers transfer jobs. Scheduled job logs are routed through the same channels but may need additional handler integration in scheduledJobWorker.
