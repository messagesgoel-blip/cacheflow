## Changes

### `web/lib/auth/requestAuth.ts` — `decodeAuthPayload`

**Vulnerability fixed:** Previous implementation fell back to `jsonwebtoken.decode()` (no signature check) when `JWT_SECRET` was absent or when `verify()` threw. This allowed any attacker-crafted JWT to authenticate successfully.

**Fix:** `JWT_SECRET` is now required. If absent, `decodeAuthPayload` returns `null` (unauthenticated) and logs an error. `decode()` is no longer called anywhere in the auth path.

#### Signature (unchanged — no breaking API change)

```ts
export async function decodeAuthPayload(token: string): Promise<AuthPayload | null>

export interface AuthPayload {
  id?: string | number;
  email?: string;
}
```

#### Error behaviour

| Condition | Return value | Side effect |
|-----------|-------------|-------------|
| `JWT_SECRET` not set | `null` | `console.error` |
| Token signature invalid | `null` | none |
| Token expired | `null` | none |
| Token malformed | `null` | none |
| Token valid | `AuthPayload` | none |

### `web/middleware.ts` — Next.js Route Guard

Edge middleware that enforces authentication at the routing layer.

**Behaviour by route class:**

| Route pattern | Auth required | Unauthenticated response |
|---------------|--------------|--------------------------|
| `/api/auth/**` | No | — (pass through) |
| `/auth/**`, `/login`, `/register`, `/health` | No | — (pass through) |
| `/api/**` (all other) | Yes — `accessToken` cookie | `401 JSON` |
| `/dashboard`, `/files`, `/connections`, `/settings`, `/remotes`, `/admin`, `/security`, `/providers`, `/conflicts`, `/share` | Yes — `accessToken` cookie | `302 → /?mode=login` |
| Everything else | No | — (pass through) |

**401 response shape (protected API routes):**

```json
{ "success": false, "error": "Authentication required" }
```

**Redirect target (protected page routes):**

```
Location: /?mode=login
```

#### Edge cases

- Middleware checks **cookie presence only** (`accessToken`). Full JWT signature verification is done inside each API handler via `decodeAuthPayload()`.
- `/_next/static`, `/_next/image`, `favicon.ico` are excluded from the matcher and never reach the middleware.
- A missing `JWT_SECRET` does not affect middleware (edge runtime cannot use Node `jsonwebtoken`); it fails gracefully inside API handlers.

#### Example payloads

**Unauthenticated API call:**
```
GET /api/connections
Cookie: (none)

HTTP/1.1 401
Content-Type: application/json
{"success":false,"error":"Authentication required"}
```

**Unauthenticated page visit:**
```
GET /dashboard
Cookie: (none)

HTTP/1.1 302
Location: /?mode=login
```

**Authenticated request (passes through):**
```
GET /api/connections
Cookie: accessToken=<jwt>

HTTP/1.1 200
...
```

## Client-visible behaviour notes

Authenticated-success API response shapes are unchanged. However, unauthenticated requests now receive new responses that did not exist before:

- Protected API routes (`/api/**` except `/api/auth/**`) return `401 JSON` (`{"success":false,"error":"Authentication required"}`).
- Protected page routes (`/dashboard`, `/files`, etc.) return `302` redirect to `/?mode=login`.
