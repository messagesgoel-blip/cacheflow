# Contract: 2.13
## Endpoint or File Path
- `web/app/api/auth/2fa/setup/route.ts` (`POST /api/auth/2fa/setup`)
- `web/app/api/auth/2fa/verify/route.ts` (`POST /api/auth/2fa/verify`)
- `web/app/api/auth/2fa/status/route.ts` (`GET /api/auth/2fa/status`)
- `web/app/api/auth/2fa/disable/route.ts` (`POST /api/auth/2fa/disable`)
- `web/lib/auth/totp.ts`

## Request / Response Shape
- Setup (`POST /api/auth/2fa/setup`)
  - Request: authenticated user (access token required)
  - Response: `{ success: true, qrCodeUrl, backupCodes }`
  - Sets cookies: `totpSecret`, `totpBackupHashes`
- Verify (`POST /api/auth/2fa/verify`)
  - Request body: `{ code?: string, secret?: string, backupCode?: string }`
  - Response: `{ success: boolean, sessionToken?: string, error?: string, requiresBackup?: boolean }`
  - Sets/updates cookies: `totpEnabled`, `totpLastUsed`, `totpSecret`, `totpBackupHashes`
- Status (`GET /api/auth/2fa/status`)
  - Response: `{ enabled: boolean, lastUsed?: string, backupCodesRemaining?: number, error?: string }`
- Disable (`POST /api/auth/2fa/disable`)
  - Request body: `{ password: string }`
  - Response: `{ success: boolean, error?: string }`

## Error Codes
- `401`: missing/invalid authentication; invalid verification code.
- `400`: missing required fields (code/backup code/password/secret as applicable).
- `500`: setup/verify/disable/status internal errors.

## Edge Cases
- TOTP verification allows adjacent time windows (`window=1`) to handle clock skew.
- Backup codes are hashed for storage and consumed on successful backup-code login.
- Setup issues short-lived cookies first, then verify promotes to longer-lived 2FA cookies.

## Example Payload
```json
{
  "code": "123456"
}
```
