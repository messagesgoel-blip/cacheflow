# Contract 4.8: Share Link Proxy

## Endpoint
`GET /s/[linkId]`

## Purpose
This endpoint acts as a proxy for share links, hiding the underlying provider by redirecting users to the actual destination URL without exposing the original source.

## Request
- Method: `GET`
- Path: `/s/{linkId}`
- Parameters:
  - `linkId` (path parameter): Alphanumeric string with optional dashes and underscores, representing the unique identifier for the shared link

## Response
The endpoint performs an HTTP redirect to the destination URL associated with the linkId.

### Success Response
- Status Code: `302 Found`
- Headers:
  - `Location`: The destination URL to redirect to
  - `Cache-Control`: `no-cache, no-store, must-revalidate`
  - `Pragma`: `no-cache`
  - `Expires`: `0`

### Error Responses
- `400 Bad Request`: Invalid link ID format
  - Body: `{ "error": "Invalid link ID format" }`
  
- `404 Not Found`: Share link not found
  - Body: `{ "error": "Share link not found" }`
  
- `500 Internal Server Error`: Server-side error occurred
  - Body: `{ "error": "Internal server error" }`

## Edge Cases
- Invalid linkId format (contains special characters other than dash/underscore)
- Expired or deleted share links
- Database connectivity issues
- Malformed destination URLs

## Example Payload
Request: `GET /s/a1b2c3d4-e5f6-7890`
Response: `302 Redirect` to `https://example.com/shared/a1b2c3d4-e5f6-7890`

## Security Considerations
- Input validation on linkId to prevent injection attacks
- Proper sanitization of destination URLs to prevent open redirect vulnerabilities
- Rate limiting to prevent abuse of the endpoint
- Cache headers to prevent unwanted caching of redirects

## Implementation Notes
- The linkId is validated using a regex pattern: `/^[a-zA-Z0-9_-]+$/`
- The actual destination URL is retrieved from the database based on the linkId
- The endpoint uses a temporary redirect (302) to maintain flexibility for future changes
