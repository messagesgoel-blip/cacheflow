### Remote URL Import (RIMPORT-1)

**Files changed:**

| File | Change |
|------|--------|
| `lib/transfers/remoteUpload.ts` | Added SSRF protection (`validateRemoteUrl`), file size limit (10 GB), extended timeout to 5 min, pre- and post-download size checks |
| `api/src/routes/remoteUpload.js` | Added auth middleware, provider allowlist validation, delegates URL validation to service layer |
| `api/src/services/remoteUploadService.js` | Added `validateRemoteUrl` with private IP blocking, size limits, exported for route-level use |

**Security measures:**
- Private/internal IP blocking (localhost, 10.x, 172.16-31.x, 192.168.x, link-local, IPv6 loopback/ULA)
- DNS resolution validation (`validateResolvedIPs`) — resolves hostname and checks all A/AAAA records against private IP ranges before connecting; normalizes `::ffff:` IPv4-mapped IPv6 addresses before checking
- Manual redirect handling (`redirect: 'manual'`) — resolves relative `Location` headers to absolute URLs first, then validates and checks resolved IPs to prevent SSRF via open redirects
- Streaming download with incremental size enforcement — body is consumed as a stream with byte counting; never materializes full file before size check
- Protocol restricted to HTTP/HTTPS only
- File size enforced both from Content-Length header and actual download size
- Auth middleware required on all endpoints
- Provider allowlist: google, onedrive, dropbox, box, pcloud, filen, yandex, vps, webdav, local
- Both TS (`lib/transfers/remoteUpload.ts`) and JS (`api/src/services/remoteUploadService.js`) layers share aligned SSRF protections

**Transfer route contract:**
- `GET /progress/:taskId` — returns `{success, taskId, progress, status}` or 404; ownership-gated to `req.user.id`
- `POST /start` — requires `sourceProvider`, `destProvider`, `fileId`; persists `owner` from `req.user.id`; returns `{success, taskId}` with 201
- `DELETE /:taskId` — removes transfer or returns 404; ownership-gated to `req.user.id`

**Note:** POST /start enqueues a real BullMQ transfer job via `addTransferJob()` from `api/src/services/transferService.js` (a JS wrapper that delegates to the same BullMQ queue as `lib/queue/transferQueue.ts`) and stores the BullMQ job ID in the local registry. DELETE cancels the BullMQ job via `cancelTransferJob()`. The in-memory `Map` is process-local — suitable for single-instance or sticky-session deployments; use Redis-backed state for horizontal scaling.

### Task 6.2 — Real-Time Terminal + Log View (Enhanced)

**Files added:**

| File | Change |
|------|--------|
| `web/components/files/JobLogPanel.tsx` | Added terminal-like component that streams job progress over SSE |

**Implementation notes:**

- Connects to `/api/jobs/logs?jobId=...` as the primary SSE endpoint
- Displays real-time log entries with timestamps, severity levels (info/warn/error/success)
- Auto-scrolls to show latest logs
- Shows connection status and progress percentage
- Can be opened for any active transfer job
- The stream emits `connected`, `log`, `progress`, and terminal `done` events; `done` closes the stream after `completed` or `failed`
- `/api/transfers/[id]/progress` remains a legacy fallback path, not the documented primary contract

**SSE Contract: `/api/jobs/logs`**

**Request**

```http
GET /api/jobs/logs?jobId={jobId} HTTP/1.1
Cookie: accessToken={httpOnly-cookie}
Accept: text/event-stream
```

**Response headers**

```http
Content-Type: text/event-stream
Cache-Control: no-cache, no-transform
X-Accel-Buffering: no
```

**Event examples**

```text
event: connected
data: {"jobId":"job-123","userId":"user-456"}

event: log
data: {"jobId":"job-123","jobType":"transfer","userId":"user-456","level":"info","message":"Starting transfer","timestamp":1700000000000}

event: progress
data: {"jobId":"job-123","jobType":"transfer","userId":"user-456","progress":50,"status":"progress","timestamp":1700000005000,"data":{"fileName":"report.zip","operation":"copy"}}

event: done
data: {"status":"completed"}
```

**Field notes**

- `timestamp` is epoch milliseconds from the worker/bridge payload.
- `level` is one of `info`, `warn`, `error`, `debug`; `JobLogPanel` maps `debug` to info styling.
- `message` is a plain-text worker log line.
- `progress` may be numeric or structured; `JobLogPanel` only renders percentages for numeric values.
- `done.status` is `completed` or `failed`; `done.error` is present on failed terminal events.

**Error and completion semantics**

- Pre-stream failures return HTTP status codes such as `400`, `401`, or `500`; no SSE stream starts in those cases.
- Runtime connection problems surface via the browser `EventSource.onerror` path; the panel logs a reconnect message and clears the connected indicator.
- `done` is terminal. On `completed` or `failed`, the server emits one `done` event and then closes the stream.
